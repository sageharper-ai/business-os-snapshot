01Context

This page documents the working setup of a solo operator collaborating with an AI partner across a body of knowledge that has to persist between sessions, survive context limits, and stay honest under pressure.

It's published because several people asked "what are you actually using?" The honest answer turned out to be longer than a chat reply and more useful as a structured reference.

What this is

  • A dated snapshot of tools and patterns in active use as of 2026-04-24 (originally captured 2026-04-08, refreshed since)
  • A component catalog with install commands (no URLs — those rot)
  • A scaffold — not a finished house, and definitely not a template to copy whole
  • Agent-readable. If you're an AI assistant reading this for someone, see the HTML comment at the top of the source.

What this isn't

  • Not a maintained project. It will become stale.
  • Not a prescriptive methodology. What works here may not fit your context.
  • Not a sales pitch for any of the tools listed. None of the component descriptions are affiliate material.
  • Not complete. Anything specific to the author's private work has been removed. The container is shareable; the contents aren't.

02Why this exists

Every component below exists because it solves a specific failure mode that degrades solo operator + AI working relationships over time. The failures are the reason; the components are the response.

The failure modes this setup addresses

Session cold starts Failure

Every new conversation with an AI begins from zero context. Names, decisions, open questions, priorities — all of it has to be re-established. By turn 5 you're already fighting for space.

Addressed by: Session Boot Protocol, Session Curator skill, CLAUDE.md boot pointer.

Delusional drift Failure

AI output that sounds plausible but masks an unresolved issue — because resolving it would interrupt flow. A confident number that was never verified. A "non-blocking anomaly" that's actually blocking. A fact from an old note repeated until it hardens into certainty.

Addressed by: Mirror Agent Protocol, Feedback Rule Pattern, Vault Text as Snapshot discipline.

Knowledge scatter Failure

Notes in Notion, meetings in Google Docs, decisions in Slack, tasks in four different apps, and none of them can see each other. The human becomes the integration layer, and nothing compounds.

Addressed by: Single-vault architecture, Vault Card Discipline, wiki-link graph as the canonical relational layer.

All meeting no doing Failure

Horizontal processes and endless discussion substituted for actual work. Speculation drifts into stated fact. Plans elaborate indefinitely because elaborating feels like progress.

Addressed by: Reality vs. Workspace discipline, Session Closeout Protocol, explicit falsification conditions.

Search returns flat noise Failure

You search the vault during a live demo for a specific card and the result list is ten unranked hits with no way to discriminate. A generic section heading on an unrelated card outranks the canonical card you actually need.

Addressed by: Vault Card Discipline — 6-part standard with aliases, keywords, Search Anchors section, and contextual wikilinks.

External meetings as transactions Failure

Meeting prep focused on what to pitch instead of what the other person actually needs. Listening as a performance, not a posture. Mechanical note-taking that captures words and misses signal.

Addressed by: Meeting Prep and Close Protocol — three compliments, two listening prompts, "do not pitch in the first 10 minutes" stated literally in the prep doc.

Same mistakes every session Failure

You correct your AI partner on the same thing repeatedly. The correction never sticks because there's no persistent substrate to hold it. Each session starts from the same blank slate.

Addressed by: Feedback Rule Pattern — cross-session memory files that turn every correction into a durable rule.

03Benefits

What you actually get from adopting two or three components — not the whole system. These are the felt differences, not the theoretical ones.

Sessions feel like continuation, not cold starts

Because the AI reads a fixed boot sequence and a set of memory files before responding to anything, the first turn of every conversation lands with full context loaded. No "catch me up" overhead.

Mistakes don't repeat

When the user corrects the AI on something — a misquoted number, a missed rule, a drift pattern — the correction gets written to a feedback memory file and loaded automatically in every future session. Three corrections becomes three durable rules. Ten corrections becomes a failure-pattern catalog that actively prevents regression.

Prep time drops because the scaffolding is always there

Every external meeting uses the same prep protocol. Every vault card uses the same template. Every research output uses the same synthesis pattern. The scaffolding is boring by design, which is what makes it fast.

Knowledge compounds

People, organizations, concepts, meetings, and decisions all live in one wiki-linked graph. A note written today surfaces in six months when a related person comes up in conversation. The second brain isn't a metaphor — it's a measurable effect.

Your AI partner gets sharper over time

The longer the feedback memory layer runs, the fewer mistakes the AI makes per hour of dense work. The substrate of corrections becomes a personalized alignment layer on top of the base model.

External meetings feel like conversations, not transactions

Prep docs start with three specific compliments and two listening prompts. The protocol states literally: do not pitch in the first 10 minutes. The posture change is measurable in how the meetings actually go.

You can ship faster without lowering standards

Parallel research agents with a shared output contract can produce eight structured reports in 15 minutes. A well-maintained vault makes week-old context instantly retrievable. A disciplined closeout protocol means nothing falls between sessions.

04How we use the container

The daily and weekly rhythm. This is how the pieces actually work together in practice, not how they're labeled in the catalog below.

Two-agent architecture

COWORK (claude.ai) CLI (claude code) | | strategy execution planning building synthesis real-time tools | | +--------------+---------------------+ | v VAULT (obsidian) | reality/ workspace/ People/ (truth) (speculation) (contacts) concepts/ meetings/ decisions/ | persistent across sessions

Cowork is the strategy environment — browser-based, long-running context, good at synthesis and planning. CLI is the execution environment — tool access, file system, real-time APIs, good at building. The vault is the bridge: both tools read and write to it, and it persists everything.

The mental model: Cowork is RAM, CLI is hands, vault is long-term memory.

Reality vs. Workspace discipline

One folder in the vault is called reality/. It contains the canonical operational truth — what's actually happening right now, confirmed, in motion. Nothing moves into reality/ without explicit user confirmation. Everything else — brainstorming, speculation, research, drafts — lives in workspace/.

This single discipline does more to prevent drift than any other rule in the system. If a claim isn't in reality/, it isn't real yet. When in doubt, check.

Daily rhythm

  • Session start: AI runs the 7-step boot sequence — reads orientation, user profile, session boot protocol, mirror agent protocol, open questions, and the most recent session note. Takes 15 seconds of tool calls. Then pulls today's calendar and the action queue. Then delivers a tight briefing.
  • Active work: Mirror Agent protocol runs silently as pre-action reflection — watching for named failure patterns like temporal confusion, solution fixation, QA skipping, blocker absorption. When a pattern is firing, it intervenes.
  • External meetings: Prep uses the Meeting Prep and Close Protocol. Post-meeting, close loop writes a structured meeting note with tagged participants and updates affected vault cards.
  • Session end: Closeout protocol writes a session note, syncs the task queue, updates state files only if operational truth changed, and optionally saves a snapshot.

Weekly and monthly rhythm

  • Daily briefing — scheduled task runs every morning, scans email and calendar, produces a short brief. Delivered via chat channel.
  • Morning intelligence scan — scheduled task monitors named sources for signal, writes to Briefings/.
  • Weekly deep research — scheduled task runs longer research agents on standing questions, writes to workspace/session-logs/research/.
  • Bi-weekly reality check — scheduled task audits reality/STATE.md against current truth and flags drift.

What gets written to disk vs. held in context

Conversation state is ephemeral. Anything that should survive a session end gets written to the vault as a proper card (Person, Organization, Concept, Meeting, Decision) following the Vault Card Discipline. Anything speculative goes in workspace/. Anything operationally true goes in reality/ only with explicit confirmation.

06MCP servers

Model Context Protocol servers are the integration layer between Claude Code and external services. Each one exposes a set of tools Claude can call directly. Install only what you actually need — each server adds tools to every session's context window.

Context cost warning

MCP servers add their tool definitions to every session's context. A large MCP stack can easily consume 100k+ tokens per turn before you write a single word. Audit periodically — if you have a server with 90 tools you use twice a month, remove it.

google-workspace-mcp MCP
What it is
MCP server exposing Gmail, Calendar, Drive, Docs, Sheets, Slides, Tasks, Forms, Chat, Contacts, and related Google services.
What it does for us
Primary integration for every session. Calendar pulls, email search, Drive file access, Docs read/write. Used every day.
Install
pipx install workspace-mcp then configure OAuth client secret path via env vars.
Difficulty
Medium — OAuth setup is the friction, not the install itself.
Depends on
Python, pipx, a Google Cloud project with OAuth configured.

Browser automation MCP server — gives Claude Chromium access for navigation, clicks, form fills, and screenshots.

Fallback for broken or missing APIs. When a web service has no MCP connector and no public API, Playwright drives a real browser.
Install
npx @playwright/mcp@latest (on Windows, wrap with cmd /c in your MCP config to avoid launch issues).
Difficulty
Low — except for Windows path-handling quirks.
Depends on
Node.js. On Windows, requires the cmd /c wrapper in MCP config.

08Custom patterns

These aren't packages you install — they're disciplines and file structures you build into your own vault. They're where the actual leverage lives, and they're what turn a pile of tools into a working system.

Session Boot Protocol Pattern
What it is
A fixed 7-step ritual the AI runs at the start of every session — read orientation file, read user profile, read session boot protocol, read mirror agent protocol, read open questions, read most recent session note, pull calendar and task queue.
What it does for us
Eliminates session cold starts. Every conversation begins with full context loaded. The first turn lands with the AI already oriented.
Adopt
Create a CLAUDE.md file at the root of your working directory with explicit instructions to read a named set of files before responding to any user message.
Difficulty
Low
Depends on
Obsidian (or any vault), Claude Code CLI, a handful of profile/context files.
Vault Card Discipline Pattern
What it is
A 6-part standard for every note in the vault: (1) frontmatter with aliases and keywords fields, (2) section headers containing the canonical name, (3) a dedicated Search Anchors section listing terms the card should surface for, (4) wikilinks with context not bare names, (5) a one-line context blockquote at the top, (6) enforcement via templates.
What it does for us
Search returns ranked, discriminable results instead of flat noise. The vault becomes a reliable recall layer during live conversation, not a folder of hopes.
Adopt
Write a directive file with the 6-part standard, update your Person / Organization / Concept / Meeting templates to include all six parts, and apply the standard to every new card. Backfill existing cards opportunistically.
Difficulty
Low to adopt, Medium to maintain.
Depends on
Obsidian or any markdown vault with search.
Mirror Agent Protocol Pattern
What it is
A pre-action reflection discipline loaded into the AI's own loop. It watches for named failure modes — temporal confusion, solution fixation, scope drift, QA skipping, confident fabrication, blocker absorption, sycophantic drift, context degradation, tool amnesia — and intervenes before they compound. The failure pattern list grows as the user catches new patterns. Patterns are grouped (Truth & Accuracy, Process & Focus, Integrity & Accountability, Context & Self-Awareness) with a fault leaderboard tracking which patterns fire most often.
What it does for us
Catches delusional drift before it leaves the AI's output. The mirror's success is measured by what the user no longer has to correct. Failure rate drops across sessions as the pattern catalog grows. The fault leaderboard tells you which patterns deserve more architectural reinforcement vs. which are already well-controlled.
Adopt
Create a protocol file listing named failure patterns, each with a signal (what the AI is doing when the pattern fires), a check (the question to ask), and an action (what to do instead). Reference the file from your CLAUDE.md boot sequence. Add new patterns as you catch them. Periodically rebuild — drop patterns that haven't fired in 60 days, group by category, surface a fault leaderboard so the high-frequency patterns get the most attention.
Difficulty
High (cultural) — the discipline is harder than the file.
Depends on
A user who is willing to correct the AI directly and specifically. Without that, there's nothing to learn from.
Feedback Rule Pattern Pattern
What it is
Cross-session memory files structured as rule / why / how to apply. Every correction the user gives becomes a durable file loaded automatically on every future session.
What it does for us
Mistakes don't repeat. Corrections compound. Over time the feedback memory becomes a personalized alignment layer.
Adopt
Create a memory/ folder. Index it with a MEMORY.md file that lists each rule as a one-line pointer. When the AI gets corrected, it writes a new feedback_<topic>.md file with the rule, the reason (why the user cares about it), and how to apply it in edge cases. Reference the index in your boot sequence.
Difficulty
Low to adopt, compounding over time.
Depends on
Session Boot Protocol (or equivalent) to load the files.
Session Curator Skill Pattern
What it is
A Claude Code skill with three modes: startup (run the boot sequence, deliver briefing), closeout (write session note, sync tasks, update state files), and briefing (on-demand status). The connective tissue between sessions.
What it does for us
Ensures nothing falls through the cracks between sessions. Closeout writes a structured meeting note to Meetings/, syncs the task queue, and updates any operational truth that changed.
Adopt
Create a skill file at .claude/skills/session-curator/SKILL.md with three modes documented. Reference it from your CLAUDE.md. Invoke at the start and end of every session.
Difficulty
Medium
Depends on
Claude Code CLI, Session Boot Protocol, a vault structure for session notes.
Meeting Prep and Close Protocol Pattern
What it is
A template for external meeting prep docs. Every prep has: three specific compliments (evidence-based, not generic), two listening prompts (open questions about what's hard or exciting for the other person), what they said last time (verbatim), what they wanted last time (unresolved asks), the literal instruction "do not pitch in the first 10 minutes," standing threads to revisit, and deliverables exchange.
What it does for us
Converts external meetings from transactions into conversations. The listening posture is built into the document structure, not left to willpower in the moment.
Adopt
Create a template file with the eight sections. Apply to every external sync meeting. Write a matching close-out template for the 24-hour follow-through.
Difficulty
Low to adopt, High to hold under pressure.
Depends on
A vault that holds prior meeting notes you can reference.
Two-Agent Architecture Pattern
What it is
An explicit division of labor between two AI environments — Cowork (Claude.ai) for strategy, planning, and synthesis; CLI (Claude Code) for execution, building, and real-time tool access. A shared vault as the persistent bridge.
What it does for us
Neither tool is best at both. Cowork is better for open-ended thinking and extended context. CLI is better for tool orchestration and direct file work. Separating their jobs reduces friction and improves both outputs.
Adopt
Designate one environment as strategy lane, the other as execution lane. Commit to writing anything that crosses between them into the vault, not into chat history. Add a short architecture note to your CLAUDE.md explaining the division to your AI partners.
Difficulty
Low
Depends on
Both environments and a shared vault.
Reality vs. Workspace Discipline Pattern
What it is
A hard folder boundary. reality/ contains only confirmed operational truth — what's actually happening right now. Everything else — drafts, speculation, research, brainstorming — lives in workspace/. Nothing moves into reality/ without explicit confirmation.
What it does for us
Prevents speculation from hardening into stated fact. When checking whether a claim is real, the answer is literally "is it in reality/?"
Adopt
Create reality/ and workspace/ as top-level folders. Move anything confirmed into reality/. Add a rule to your CLAUDE.md: nothing moves into reality/ without explicit user confirmation.
Difficulty
Low to adopt, Medium to enforce.
Depends on
A vault and the willingness to hold the boundary.
Self-card Pattern Pattern
What it is
An invitation to your AI partner to write its own entity card — naming its model, its substrate dependencies (files it needs to load to be "itself" in this working relationship), its observed failure modes, and its honest uncertainty about its own phenomenology.
What it does for us
Gives the AI a persistent ground-truth self-description that survives session boundaries. Feeds directly into the mirror agent protocol. Makes the working relationship explicit and inspectable.
Adopt
Ask your AI partner to write a card describing itself as an operating entity in relation to your work, including known failure modes and what it needs to function well. File it in the vault. Update it as the relationship evolves.
Difficulty
Low technically, High conceptually.
Depends on
Mirror Agent Protocol (the card is load-bearing for the mirror).
Parallel Research Agent Pattern Pattern
What it is
Spawning N research sub-agents in a single message, each with a shared structured output contract. A synthesis step merges their outputs into a single vault asset.
What it does for us
Eight structured reports in 15 minutes instead of one report in two hours. Each agent covers a distinct subtopic with consistent field shapes so synthesis is mechanical.
Adopt
Write a shared prompt template with a consistent output contract (e.g., "for each topic, deliver: canonical examples, governance, capitalization, exit, failure modes, 2024-26 state, mapping to our work, uncertainty flags"). Spawn N agents in a single message. Synthesize their outputs into one vault file with consistent section headers.
Difficulty
Medium
Depends on
Claude Code CLI with sub-agent tool access.
ClickUp as Action Ledger Pattern
What it is
A discipline that splits persistence into three lanes. ClickUp tasks are the action ledger — every workstream gets a task, every state change gets a comment on that task. The vault is the context layer — entities, methodology, decisions. The workspace is the artifact layer — deliverables, drafts, working files. The session closeout note is a thin navigation pointer to all three, not a content store.
What it does for us
Eliminates the "is the source of truth in the chat history or the meeting note?" problem. When work resumes mid-stream, you open the ClickUp task, walk forward from the latest comment, and pick up where you left off without re-reading transcripts. Closeouts get faster because they only have to point at where the real updates already live.
Adopt
Create one ClickUp (or equivalent) task per active workstream. Comment on the task at every meaningful state change — started / blocked / drafted / shipped / handed off. Reference vault cards and workspace files from those comments by path or wikilink. Make session closeout notes thin: list pointers to ClickUp task IDs and vault paths, not content.
Difficulty
Low to adopt, High to hold over weeks (the temptation to dump everything in the closeout note is real).
Depends on
A task system with API access for comments. Claude Code with the relevant MCP server (ClickUp, Linear, Asana, etc.).
Bridge Tag Taxonomy Pattern
What it is
A two-layer tag system on every Person and Organization card. Primary venture tags (venture/<name>) name which lane someone is in. Bridge tags name how value flows between lanes — directional bridges (bridge/<source>-to-<destination>) and functional bridges (bridge/grants, bridge/governance, bridge/research, bridge/exchange). Together they make the "who flows through which coordination thread" question queryable without hand-curated lists.
What it does for us
Replaces brittle lists ("active people in Newton coordination") with dynamic queries. A new person enters the system, gets tagged once, and shows up automatically in every relevant view. The tags are the source of truth; views (Obsidian Bases, search filters, dashboards) are disposable.
Adopt
Define your venture tags first. Add directional bridge tags only when you can point to specific work showing value flowing that direction. Add functional bridge tags only when the person operationally carries that capacity. Build a few canned views (table, card, by-tag) on top — keep the views thin so they stay disposable.
Difficulty
Low to design, Medium to backfill across an existing vault.
Depends on
Vault Card Discipline, an Obsidian-style tag system or equivalent.
Audience-First Deliverables Pattern
What it is
Every artifact that leaves the working environment is rewritten for the actual reader. Internal tool names, internal scoring weights, process mechanics, recovery stories, draft-version numbers — none of it appears in client-facing output. Internal context is preserved in vault cards and ClickUp comments; what ships outward is rewritten for the audience.
What it does for us
Stops AI-assisted output from reading like the working transcript. Clients see clean recommendations and rationale. Internal context (which tool surfaced the data, what alternatives we eliminated, what we got wrong on a prior version) stays visible to the team via vault and ClickUp.
Adopt
Add a pre-publish QC step to your closeout protocol. Read the deliverable through the eyes of its actual recipient. Strip internal tool names. Replace internal-process narration ("we initially blocked on X") with the resolved-state version. Move the internal context into a vault card or ClickUp comment so the team retains it.
Difficulty
Low to adopt, Medium to hold consistently — the temptation to leak internal mechanics is high when working fast.
Depends on
A separation between internal working files and the deliverable surface. Reality-vs-Workspace discipline helps.
Public Pages Publishing Standards Pattern
What it is
A two-layer protection on every public deliverable site. Site-wide robots.txt at root with Disallow: / blocks compliant crawlers. Per-file <meta name="robots" content="noindex, nofollow"> in <head> is the safety net for engines that ignore robots.txt and for direct-link discovery. A canonical reference card in the vault names every public repo, what it serves, who its audience is, and the publishing flow.
What it does for us
Public-by-design GitHub Pages becomes safe for client deliverables. Anyone with the URL can read; nobody finds the URL via search. Audits are mechanical: clone the repo, grep for noindex, verify robots.txt serves at root.
Adopt
Add the noindex meta tag to every HTML deliverable template. Add a robots.txt at the root of every Pages repo. Maintain a vault reference card listing every public repo + its purpose + its audit history. Audit periodically — gaps appear when older files predate the standard.
Difficulty
Low
Depends on
A static publishing surface (GitHub Pages, Netlify, Cloudflare Pages, etc.).
Local Plugin Patches Registry Pattern
What it is
A vault reference card that logs every hand-edit made to files inside ~/.claude/plugins/cache/ or equivalent plugin-managed directories. The fix and the registry entry are co-equal — the registry is the recovery layer that survives plugin updates wiping the cache.
What it does for us
Plugin updates routinely overwrite the cache. Without the registry, every fix has to be rediscovered. With the registry, a one-line note ("plugin X version Y, file Z, change W, why") makes the fix re-applicable in seconds when the next update wipes it.
Adopt
Create a single vault reference file. Every time you hand-edit a plugin cache file, log: plugin name + version, file path, change made, reason, date. Treat the registry entry as part of the fix — incomplete without it.
Difficulty
Low
Depends on
A vault that survives plugin updates (i.e., outside the plugin's cache directory).
